TechHQ embraces a collaborative model inspired by wikis and open source communities. In our case, we follow an
inner source
approach, where everyone is
encouraged to share, improve, and build together across teams.
What should you put on TechHQ?
Technical guidance on TechHQ should be sufficiently complete to aid architects and detailed enough for AI usage. To keep
our technical knowledge DRY (Don't Repeat Yourself), avoid repeating content already published elsewhere; instead, link
to important external resources.

Source Code Contributions
â€‹
Minor, individual file edits
can be made directly to the dev branch. These edits will be automatically deployed to
https://techhq-d.dc.lilly.com
.
All edits
will be reviewed prior to prod deployments. Reviewers will vary based on
the change. Commit messages must capture the change rationale.
For new docs drafts and multi-doc edits
, create a branch with the prefix
draft/
. Open a PR to dev when ready.
For new site features
, create a branch with the prefix
features/
. Open a PR to dev when ready.
Neither draft/ nor feature/ branches are automatically deployed. 'Automatically delete head branches' is not enabled;
delete your branch if it is complete or inactive.
