Instrumentation
Instrumentation is the process of adding monitoring capabilities to your applications to collect metrics, traces, logs, and frontend performance data. Our observability stack leverages
Alloy
, an open-source collection framework, to receive and process telemetry data from your applications. This guide covers how to instrument your applications to send data to the CATS observability platform.
Overview
​
The CATS platform uses
Alloy
as a centralized data collection and relay service that supports multiple telemetry protocols:
Prometheus Metrics
: Remote write protocol on port 9999
OpenTelemetry (OTEL)
: Both gRPC and HTTP endpoints for traces and metrics
Faro
: Frontend monitoring and browser analytics
Loki
: Structured logging aggregation

Frontend Monitoring with Faro
​
Faro SDK Overview
​
Faro is Grafana's frontend monitoring solution that collects browser performance data, errors, and user interactions. The Faro receiver is exposed on port 12347.
Faro SDK Installation
​
JavaScript/TypeScript:
npm install @grafana/faro-web-sdk

Frontend Performance Metrics
​
Faro automatically collects:
Page Performance
: Largest contentful paint (LCP), First input delay (FID), Cumulative layout shift (CLS)
Web Vitals
: Core web vitals metrics
Errors
: JavaScript errors, unhandled promise rejections, and console errors
User Interactions
: Click events, navigation timing
Session Information
: User sessions, page views, session duration
Frontend Logs
​
Frontend logs are collected and labeled with:
cluster_name
: Your cluster environment
region
: AWS region
source
: Set to "faro" for easy filtering
Access frontend logs in Grafana's Loki dashboard and filter by
source="faro"

Alloy Configuration Details
​
The Alloy service is configured with the following components:
Logging
​
Alloy's internal logs are configured at ERROR level and sent to Loki with appropriate labels:
instance
: Hostname
job
: integrations/self
cluster_name
: Your cluster name
region
: AWS region
account_name
: AWS account name
Prometheus Metrics Pipeline
​
Receiver
: Accepts Prometheus remote write on port 9999
Relabeling
: Adds cluster name and region labels
Remote Write
: Forwards to Prometheus at
prometheus.monitoring:9090
OTEL Pipeline
​
Receiver
: OTLP receiver listening on gRPC (4317) and HTTP (4318)
Processor
: Batch processor for efficient delivery
Exporters
:
Prometheus exporter for metrics
Jaeger exporter for traces (at
jaeger-deployment-collector.logging
)
Faro Pipeline
​
Receiver
: Faro receiver on port 12347
CORS
: Configured to accept requests from all origins (
*
)
Routing
:
Logs forwarded to Loki with
source="faro"
label
Traces forwarded to batch processor and then to Jaeger
Log Processing
​
Alloy processes container logs from Kubernetes:
File Source
: Reads container logs from
/var/log/containers/*.log
Metadata Extraction
: Extracts namespace, pod, and container names from filenames
Tenancy
: Logs are organized by namespace tenant
Cleanup
: Logs older than 1 hour are dropped to control cardinality
Destination
: Loki gateway at
loki-loki-distributed-gateway.monitoring
Verification
​
Verify Metrics
​
Check if your metrics are being received:
# From your pod
curl -X POST http://grafana-alloy.monitoring:9999/api/v1/write \
-H "Content-Type: application/x-protobuf" \
-d "your-metrics-data"
# Or check Prometheus
# https://metrics.apps.lrl.lilly.com
# Query for your metric: your_app_metric_name
Verify Traces
​
Check if traces are being received:
Navigate to Grafana's Jaeger integration
Search for your application service
Verify traces appear in the trace search interface
Verify Frontend Data
​
In Grafana:
Navigate to Explore
Select Loki datasource
Filter by
source="faro"
and your app name
Verify frontend logs and performance data
Troubleshooting
​
Metrics Not Appearing
​
Verify the endpoint URL:
http://grafana-alloy.monitoring:9999
Check network connectivity between your pod and Alloy service
Ensure labels are correctly formatted
Check Alloy logs:
kubectl logs -n monitoring -l app.kubernetes.io/name=alloy -f
Traces Not Appearing
​
Verify the OTEL endpoint:
http://grafana-alloy.monitoring:4317
(gRPC) or
http://grafana-alloy.monitoring:4318
(HTTP)
Check that your SDK is configured with the correct endpoint
Verify traces are being generated (check SDK logs)
Check Alloy logs for export errors
Frontend Data Not Appearing
​
Verify Faro SDK is initialized correctly
Check browser console for errors
Verify the Faro endpoint:
http://grafana-alloy.monitoring:12347
Check CORS settings if using cross-origin requests
Ensure
sessionTracking
and
errorTracking
are enabled
High Cardinality Issues
​
Review your labeling strategy
Avoid using unbounded values (user IDs, request IDs) as labels
Use sampling to reduce trace volume
Adjust cardinality limits in Prometheus and Loki
Best Practices
​
Use Sampling
: Don't trace 100% of requests in production; use head-based or tail-based sampling
Meaningful Labels
: Use labels that represent business dimensions, not technical cardinality explosions
Structured Logging
: Use structured logging frameworks and include relevant context
Performance
: Use batch processors to group telemetry data efficiently
Security
: Never include sensitive data (passwords, tokens, PII) in telemetry
Monitoring
: Monitor your instrumentation overhead and adjust accordingly
